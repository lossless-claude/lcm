# Gitleaks Pattern Sync — Design Spec

## Goal

Replace hand-curated `BUILT_IN_PATTERNS` with 220+ auto-synced regexes from gitleaks, keeping a small set of native gap-filler patterns, with weekly automated updates that ship hotfix releases.

## Problem

`src/scrub.ts` maintains a hand-curated regex array for secret detection. After the security hardening PR this grows to ~20 patterns. This approach:

- Always lags behind new token formats
- No coverage for lesser-known providers (222 providers in gitleaks vs our 20)
- Manual maintenance burden — every new provider needs a regex PR

### Alternatives evaluated

| Approach | Coverage | Verdict |
|----------|----------|---------|
| **secretlint** (runtime dep) | 6/19 test tokens detected | Too strict — requires real key checksums/lengths |
| **gitleaks binary** (external) | Full 222 rules | Can't run in-process on strings |
| **gitleaks regex extraction** | 13/19 + native gap-fillers | Best fit — broad coverage, zero runtime deps |

secretlint's rules are designed for committed file scanning and require exact key structures (e.g., OpenAI rule checks for `T3BlbkFJ` checksum marker). Our use case — scrubbing live session transcripts — needs aggressive pattern matching where a false `[REDACTED]` is far cheaper than a leaked key.

## Architecture

### Components

```
scripts/update-gitleaks-patterns.ts   — fetch + parse + convert + test + write
src/generated-patterns.ts             — auto-generated, checked into git
src/scrub.ts                          — ScrubEngine consumes generated + native patterns
test/fixtures/token-samples.ts        — known positive/negative test cases
.github/workflows/update-patterns.yml — weekly cron sync + auto-release
```

### Data flow

```
gitleaks.toml (GitHub raw)
  → scripts/update-gitleaks-patterns.ts (fetch, parse TOML, convert Go→JS regex)
  → validate: compile each regex, run smoke tests, run coverage tests
  → src/generated-patterns.ts (checked in)
  → ScrubEngine merges [gitleaks, custom, globalUser, project]
```

## Generated patterns file

`src/generated-patterns.ts`:

```typescript
// AUTO-GENERATED by scripts/update-gitleaks-patterns.ts — do not edit manually
// Source: https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml
// Updated: 2026-03-25T00:00:00Z
// Rules: 220 (1 skipped — JS-incompatible)

export interface GitleaksPattern {
  id: string;
  regex: string;
  flags: string;  // "" or "i" (from Go's (?i) inline flag)
  description: string;
}

export const GITLEAKS_PATTERNS: GitleaksPattern[] = [
  // ... 220 entries
];
```

### Go → JS regex conversion

| Go syntax | JS equivalent |
|-----------|---------------|
| `(?i)` inline flag | Strip from regex, set `flags: "i"` |
| `\x60` | Literal backtick |
| `secretGroup = N` | Drop — we redact the full match, not a capture group |
| `(?-i:...)` | Not present in current rules; skip rule if encountered |

220/221 current gitleaks regexes are JS-compatible. The one incompatible rule (`jwt-base64`) is skipped.

### Entropy

Skipped. gitleaks uses Shannon entropy thresholds (130/222 rules) to reduce false positives on generic patterns. For transcript scrubbing, aggressive matching is preferred — a false redaction is harmless, a missed secret is not.

## Update script

`scripts/update-gitleaks-patterns.ts`:

1. **Fetch** `gitleaks.toml` from GitHub raw URL
2. **Parse** TOML — extract `[[rules]]` blocks with `id`, `regex`, `description`
3. **Convert** Go regex → JS (see conversion table above)
4. **Validate** each regex: `new RegExp(pattern)` must not throw
5. **Smoke test**: verify no pattern matches common English words ("the", "function", "import", "class", "return") — catches overly broad regexes
6. **Coverage test**: run against `test/fixtures/token-samples.ts` — known positive/negative pairs per provider. Fail if any expected positive is missed or expected negative is matched.
7. **Write** `src/generated-patterns.ts` only if all tests pass
8. **Exit** with nonzero if fetch failed, conversion produced fewer rules than expected (regression guard), or tests failed

## ScrubEngine changes

### Pattern sources (merge order)

```
1. GITLEAKS_PATTERNS[]  — 220+ auto-synced regexes
2. NATIVE_PATTERNS[]    — gap-fillers not covered by gitleaks
3. globalUserPatterns[] — from config.json (user-defined)
4. projectPatterns[]    — from sensitive-patterns.txt (user-defined)
```

### NATIVE_PATTERNS (replaces BUILT_IN_PATTERNS)

Only patterns that gitleaks does not cover:

```typescript
export const NATIVE_PATTERNS: string[] = [
  "Bearer [A-Za-z0-9\\-._~+/]+=*",
  "[Pp]assword\\s*[:=]\\s*\\S+",
  "(postgres|mysql|mongodb|redis|rediss)://\\S+:\\S+@\\S+",
];
```

All other patterns currently in `BUILT_IN_PATTERNS` (OpenAI, Anthropic, GitHub, AWS, Slack, Stripe, GCP, SendGrid, Twilio, Shopify, Vault, Doppler, JWT, npm) are now covered by gitleaks rules and are removed.

### Constructor changes

The constructor must handle flagged patterns (gitleaks patterns with `flags: "i"`):

```typescript
constructor(globalPatterns: string[], projectPatterns: string[]) {
  // Merge gitleaks (with flags) + custom + user patterns
  // For gitleaks patterns: new RegExp(pattern.regex, "g" + pattern.flags)
  // For string patterns: new RegExp(source, "g") — same as today
}
```

### Counting

For `scrubWithCounts`, the categories become:

| Category | What it includes |
|----------|------------------|
| `builtIn` | gitleaks + native patterns |
| `global` | User's config.json patterns |
| `project` | User's sensitive-patterns.txt |

This is a semantic change — `builtIn` now means "all non-user patterns" rather than "hand-curated list." The count is used internally for merge-priority ordering and in `lcm sensitive list` output.

## Doctor changes

Current behavior: shows a warning when no custom rules are set.

New behavior — informational display:

```
Secret detection
  Built-in patterns:  220 (gitleaks) + 3 (native)    ✅
  User patterns:      0 global, 2 project
  Manage patterns:    lcm sensitive add/remove
```

- Remove the "no custom rules" warning — user patterns are now an optional enhancement
- Add health check: verify `generated-patterns.ts` exists and exports a non-empty array
- Show generation date from the file header comment (regex parse the `// Updated:` line)
- Fail if generated patterns file is missing or has 0 rules

## `lcm sensitive list` changes

Current output shows `[built-in]` and `[user]` tags. New output:

```
Global patterns (config.json):
  [gitleaks]  220 patterns (synced 2026-03-25)
  [native]    Bearer [A-Za-z0-9\-._~+/]+=*
  [native]    [Pp]assword\s*[:=]\s*\S+
  [native]    (postgres|mysql|mongodb|redis|rediss)://\S+:\S+@\S+
  [user]      MY_CUSTOM_PATTERN

Project patterns (/path/to/sensitive-patterns.txt):
  (none)
```

Gitleaks patterns are summarized as a count + date rather than listing 220 regexes.

## GitHub Actions workflow

`.github/workflows/update-patterns.yml`:

```yaml
name: Sync gitleaks patterns
on:
  schedule:
    - cron: '0 6 * * 1'  # Weekly, Monday 6am UTC
  workflow_dispatch: {}    # Manual trigger

permissions:
  contents: write

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: develop
          token: ${{ secrets.PATTERNS_BOT_TOKEN }}  # PAT with branch protection bypass

      - uses: actions/setup-node@v4
        with:
          node-version: 22

      - run: npm ci
      - run: npx tsx scripts/update-gitleaks-patterns.ts

      - name: Check for changes
        id: diff
        run: |
          if git diff --quiet src/generated-patterns.ts; then
            echo "changed=false" >> "$GITHUB_OUTPUT"
          else
            echo "changed=true" >> "$GITHUB_OUTPUT"
            BEFORE=$(git show HEAD:src/generated-patterns.ts | grep -c 'id:' || echo 0)
            AFTER=$(grep -c 'id:' src/generated-patterns.ts)
            echo "before=$BEFORE" >> "$GITHUB_OUTPUT"
            echo "after=$AFTER" >> "$GITHUB_OUTPUT"
          fi

      - name: Commit and push
        if: steps.diff.outputs.changed == 'true'
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
          git add src/generated-patterns.ts
          git commit -m "chore: sync gitleaks patterns (${{ steps.diff.outputs.before }} → ${{ steps.diff.outputs.after }} rules)"
          git push

      - name: Ship hotfix release
        if: steps.diff.outputs.changed == 'true'
        run: |
          # Merge develop → main, bump patch, publish
          bash .claude/skills/lcm-release/scripts/release.sh patch \
            --message "fix: update secret detection patterns (${{ steps.diff.outputs.after }} rules)"
```

### Branch protection exception

The `develop` branch ruleset must add the bot token's identity (GitHub App or PAT user) to the "bypass actors" list. This allows the weekly sync to push directly to `develop` without a PR.

## Testing strategy

| Layer | What | How |
|-------|------|-----|
| **Generator smoke** | All 220 regexes compile | `new RegExp(r)` in update script |
| **Generator false-positive** | No match on common words | Update script checks "the", "function", etc. |
| **Generator coverage** | Known tokens detected | `token-samples.ts` fixture, checked in update script |
| **Unit tests** | ScrubEngine with gitleaks patterns | `vitest` — import generated + fixtures, assert matches |
| **Unit tests** | NATIVE_PATTERNS still work | Existing `scrub.test.ts` tests, updated for new naming |
| **Unit tests** | Doctor output format | Assert pattern counts and formatting |
| **Integration** | `lcm sensitive list` output | Assert `[gitleaks]`, `[native]`, `[user]` tags |
| **CI** | Weekly sync + release | GitHub Actions workflow |

## Migration

1. Generate initial `src/generated-patterns.ts`
2. Create `test/fixtures/token-samples.ts` with positive/negative pairs
3. Rename `BUILT_IN_PATTERNS` → `NATIVE_PATTERNS`, remove gitleaks-covered entries
4. Update `ScrubEngine` to merge gitleaks + custom + user patterns, handle `flags`
5. Update `lcm sensitive list` output formatting
6. Update doctor: remove warning, add informational display + health check
7. Update existing tests
8. Add GitHub Actions workflow
9. Update branch protection rules (manual step)
